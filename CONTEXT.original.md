# Magic MariaDB Sync

Aplikasi desktop portable untuk sinkronisasi data satu arah antar dua database MariaDB dengan skema berbeda. Tujuan utama: memindahkan data produksi ke salinan lokal untuk keperluan laporan tanpa perlu pengetahuan SQL.

## Language

**Source**:
Database MariaDB asal data — diperlakukan sebagai *system of record*.
_Avoid_: master, primary, origin, Host A (kecuali di konteks UI koneksi).

**Destination**:
Database MariaDB tujuan — *derived copy*, bukan system of record. Baris yang cocok PK dengan Source akan ditimpa saat sync.
_Avoid_: target, replica, slave, Host B (kecuali di konteks UI koneksi).

**Sync Session**:
Satu eksekusi sinkronisasi dari Source ke Destination memakai satu Mapping Profile. Punya status: `running`, `done`, `interrupted`, `failed`, atau `cancelled`. **Tidak resumable di V1** — Session yang berakhir non-`done` tidak bisa "dilanjutkan"; user yang ingin sync ulang memulai Session baru fresh dari awal. Idempotensi Source-wins UPSERT menjamin re-run pada tabel yang sudah ter-sync sebelumnya tidak menduplikasi data. Lihat ADR-0019.
_Avoid_: job, run, transfer, resume (semantik tidak ada di V1).

**Mapping Profile**:
Resep sinkronisasi yang mengikat satu pasangan Source/Destination. Berisi: Selection Set, Column Pairing per tabel, dan Rules. Disimpan sebagai JSON di kolom `mapping_profiles.rules_json`. Punya `status`: `draft` (WIP, boleh ada Column Pairing belum resolved) atau `ready` (lulus validator structural, siap dipakai Sync Session). Hanya profile `ready` yang boleh memulai Sync Session.
_Avoid_: template, config, recipe.

**Column Pairing**:
Pemetaan satu-per-satu kolom Destination ke sumber nilainya: Source column / Constant / NULL / DB Default / Skip. Auto-matched untuk kolom dengan nama identik (case-insensitive) + tipe kompatibel; sisanya user-resolved. Mapping tidak boleh punya kolom Destination yang unresolved sebelum sync mulai. Satu kolom Source boleh dipakai oleh banyak Column Pairing (fan-out 1→N); kombinasi (Destination column + Rule) yang membuat hasilnya berbeda — UI tidak memvalidasi "source sudah dipakai". **Opsi non-Source punya semantik UPSERT distinct**: `Kosongkan/NULL` = INSERT `NULL` + UPDATE `col=NULL` (selalu NULL post-sync); `Default DB` = INSERT `DEFAULT` + UPDATE `col=DEFAULT(col)` (re-sync reset ke server default); `Lewati` = kolom dihilangkan dari INSERT list + UPDATE tidak menyentuh kolom (**preserve existing value** — opt-out lokal Source-wins per kolom). Lihat ADR-0023.
_Avoid_: column mapping (terlalu generik), field binding.

**Rule**:
Aturan transformasi per Column Pairing — mengubah nilai saat sync. V1 mendukung 5 tipe whitelisted: `cast` (tipe), `enum_map` (lookup nilai), `regex_replace`, `string_op` (trim/upper/lower/substring), `date_format` (parse + reformat dengan `on_parse_error`: `null` | `keep_original_string` | `fail_row`). Maksimal satu Rule per pairing (tidak ada chaining), unconditional (tidak ada IFTTT branching meski label PRD legacy "IFTTT"). Disimpan sebagai flat JSON di `mapping_profiles.rules_json` dengan key `"<destination_table>.<destination_column>"`. Dieksekusi runtime sebagai `func(row) row`. Berbeda dari Column Pairing: pairing tentukan *dari mana* nilai datang, Rule tentukan *bagaimana* nilai diubah. PK column tidak boleh punya Rule (PK = identitas, bukan nilai — lihat Match Key).
_Avoid_: transform, mapping, IFTTT (label legacy yang menyesatkan — tidak ada conditional di V1).

**Closure Advisor**:
Mekanisme *compile-time* yang dijalankan dua kali: (1) saat user simpan Mapping Profile — closure di-ekspansi dan **Selection Set hasil ekspansi disimpan** ke `mapping_profiles.selection_json` dengan role per tabel (`user_selected` | `advisor_added`); (2) saat user klik Start — re-introspect Source, hitung ulang closure, deteksi schema drift. Closure ekspansi memakai **dual-side FK** — gabungan FK yang dideklarasikan di Source schema *dan* Destination schema (union DAG), supaya skenario "Destination punya FK yang Source tidak punya" tidak meledak runtime. Closure expand → dialog wajib (tidak membedakan asal FK ke user); shrink → info ringan, lanjut. Sync **tidak boleh** mulai sebelum Selection Set tertutup terhadap FK. Tidak menutup skenario inherently runtime (race dengan write Source live, orphan data lama) — itu di-fallback per-baris lewat error 1452.
_Avoid_: FK validator, parent checker.

**JIT Parent Sync** *(istilah PRD legacy — di V1 berarti Closure Advisor, bukan rekursi runtime)*:
Awalnya didesain sebagai rekursi runtime saat insert anak. Diputuskan untuk V1: **dipindahkan ke compile-time** sebagai Closure Advisor. Tidak ada rekursi FK saat eksekusi sync — semua tabel sudah masuk Selection Set lewat advisor.
_Avoid_: cascade insert, FK resolver, runtime parent resolution.

**Whole-table mode**:
Strategi pull tunggal di V1: `SELECT *` dari Source untuk seluruh tabel di Selection Set (yang sudah closed under FK via Closure Advisor), upsert ke Destination dalam chunked batch (500–1000 baris).

**Source-wins UPSERT**:
Kontrak penulisan: baris Destination yang Match Key-nya cocok dengan Source akan **ditimpa** dengan nilai Source. Edit lokal di Destination akan hilang. Destination diperlakukan sebagai salinan turunan, bukan sistem independen.

**Match Key**:
Tuple kolom PK Destination yang dipakai untuk identifikasi baris pada UPSERT. Closure Advisor wajib memastikan setiap kolom PK Destination di-pair ke kolom Source (bukan Constant/Default/NULL/Skip), dan **tidak boleh** di-Rule transform — PK adalah identitas, bukan nilai. AUTO_INCREMENT di Destination diabaikan saat sync; nilai eksplisit dari Source dikirim. UNIQUE-non-PK column tidak didukung sebagai Match Key di V1.
_Avoid_: primary key (ambigu — Source PK belum tentu = Destination PK), upsert key.

**Selection Set**:
Daftar tabel yang user **pilih eksplisit** untuk disinkronkan dalam suatu Mapping Profile. User memilih berdasarkan kebutuhan laporan, bukan berdasarkan topologi FK (user umumnya tidak tahu mana parent mana child). **Granularitas tabel-level only** di V1: Selection Set = `[]string` nama tabel, tidak ada kolom-subset per tabel. Filtering kolom dilakukan di lapisan Column Pairing (kolom Destination yang tidak diinginkan = "Lewati") atau di desain skema Destination itu sendiri (kolom yang tidak ada di Destination tidak akan ter-sync). Lihat ADR-0018.
_Avoid_: column selection, table+column subset.

**Dependency Closure**:
Hasil ekspansi Selection Set ke seluruh tabel induk yang dirujuk via FK (transitif). Dihitung sebelum Sync Session start, ditampilkan ke user sebagai preview, dan dieksekusi dalam topological order (induk dulu).

**Sample Preview**:
Tampilan read-only side-by-side **20 baris pertama per tabel** di Selection Set, hasil ekspansi Column Pairing + Rule diterapkan ke nilai Source — tanpa menulis apapun ke Destination. Tujuannya: user lihat hasil Rule transform sebelum klik Start, tangkap kesalahan `enum_map`, format tanggal salah, regex over-match, dll. Non-blocking (tidak wajib dibuka), tidak menahan koneksi Destination, tidak melakukan transaksi. SELECT-only ke Source. Berbeda dari Dependency Closure preview (daftar tabel) — Sample Preview adalah preview *nilai*. Bukan dry-run penuh: tidak menyentuh Destination, tidak validasi NOT NULL/unique constraint runtime — itu tetap di Preflight + chunk pertama. **Sample strategy**: `SELECT ... LIMIT 20` natural order (tanpa `ORDER BY` eksplisit) — persona target butuh sanity-check Rule transform, bukan audit kualitas data; performa cepat (early termination tanpa full scan), determinism rendah OK karena tidak load-bearing. Bukan random/hybrid sampling: edge case data ditangkap Preflight + runtime per-row fallback, bukan tugas Sample Preview.
_Avoid_: dry-run (mengaburkan batas: dry-run penuh akan tulis & rollback, Sample Preview tidak), test sync.

## Relationships

- Satu **Mapping Profile** menyimpan satu **Selection Set** + pasangan **Source**/**Destination**
- **Dependency Closure** = **Selection Set** ∪ semua induk transitif via FK
- **Closure Advisor** memaksa **Selection Set** = **Dependency Closure** sebelum Sync Session boleh start
- Satu **Sync Session** mengeksekusi **Whole-table mode** atas seluruh **Selection Set** dalam topological order (induk dulu)
- Satu **Sync Session** menghasilkan banyak **Sync Log** (per baris gagal)

## Example dialogue

> **Dev:** "Kalau user edit baris di **Destination**, lalu **Sync Session** berikutnya menyentuh PK yang sama, apa yang terjadi?"
> **Domain expert:** "Edit lokal hilang. Itu konsekuensi **Source-wins UPSERT** — **Destination** bukan tempat menyimpan perubahan, hanya salinan untuk laporan."

## Flagged ambiguities

- "sinkronisasi" sempat ambigu antara replikasi penuh, fill-gaps, dan merge — diresolusi: **Source-wins UPSERT** (Source selalu menang, Destination ditimpa).

## Operational notes

- **Error message ramah pengguna**: `internal/sync/errors.go` menyediakan `ToFriendly(mysqlErr) (userMsg, technicalMsg)`. Lookup berdasar MariaDB error code + parser regex untuk extract konteks (nama kolom, nilai, constraint) ke template Bahasa Indonesia. Whitelist V1: 1048, 1062, 1264, 1292, 1366, 1406, 1452 (row errors); 2002/2003/2006/2013 (connection errors → SSE error event, bukan row log). Fallback generic untuk kode di luar whitelist. Tidak ada i18n di V1; identifier code tetap English.
- **SSE progress stream**: `GET /api/sessions/{id}/events` kirim event `snapshot` (current counts + last 50 row failures) saat connect, lalu live stream `progress` / `row_failed` / `done`. Tidak ada replay buffer; source of truth = `sync_sessions` (counts di-update per chunk-commit, *sebelum* publish event) + `sync_logs`. Reconnect / refresh tab = snapshot baru. Header `Last-Event-ID` diabaikan di V1.
- **Profile snapshot di Sync Session**: `mapping_profiles` di-update in-place (closure expand, edit pairing) tapi `sync_sessions.profile_snapshot_json` membekukan full `{selection, rules, source_connection_id, destination_connection_id, profile_id}` saat session start. Runner membaca snapshot, bukan profile aktif — melindungi dari race mid-session dan menjamin rerun konsisten. UI Session Detail render snapshot + badge "profile sudah berubah" kalau hash snapshot ≠ hash profile aktif. Credentials tidak di-snapshot (di-decrypt runtime dari `connections`).
- **Cross-profile collision**: saat save Mapping Profile, `internal/repo/mapping_profiles.go` cek overlap Selection Set (post-Closure-Advisor) dengan profile lain yang `destination_id`-nya sama. Overlap non-kosong → hard-fail dengan `ToFriendlyCollision(conflicts)` daftar tabel + nama profile pemilik. Mencegah dua Source menulis ke tabel Destination yang sama (akan saling menimpa via Source-wins UPSERT). Self-update mengecualikan diri sendiri dari query. Tidak ada bypass V1; user yang butuh multi-Source merge ke satu tabel harus pakai dua Destination terpisah.
- **Schema drift detection**: hybrid pre-flight struktural + runtime type, **berlaku di kedua sisi (Source + Destination)**. Saat Start, `internal/sync/preflight.go` re-introspect kedua database dan diff terhadap snapshot kolom di `mapping_profiles.selection_json` (per tabel: `{name, is_pk, is_paired, side: 'source'|'destination'}`, tanpa tipe). *Blocking Destination*: paired column missing, PK column missing/berubah, table missing, kolom baru NOT NULL tanpa DEFAULT, kolom dengan opsi `Default DB` kehilangan DEFAULT-nya. *Blocking Source*: paired Source column missing (rename/drop), Source table missing — pesan "Kolom Source `{tabel}.{kolom}` tidak ditemukan. Mungkin telah di-rename atau dihapus. Buka Mapping Profile dan pilih ulang sumber kolom." *Auto-handled Destination*: kolom baru nullable, kolom baru dengan DEFAULT, type widened. *Auto-handled Source*: kolom Source baru yang tidak referenced di Pairing (tidak relevan, skip). Type narrowed (VARCHAR(100)→20, INT→SMALLINT) tidak ditangkap pre-flight V1 — dideteksi runtime per-baris lewat MariaDB error 1406/1264/1366 dan masuk `sync_logs`. **Tidak ada heuristik rename detection** (similarity match, type+position match): terlalu clever, risk salah-rebind silent — user explicit pilih ulang di Builder. Pesan drift via `ToFriendlyDrift` (jalur terpisah dari `ToFriendly` row-formatter), ditampilkan di banner pre-start. Tidak ada flag bypass di V1.
- **Retention via capacity, bukan waktu**: V1 tidak punya batas hari — yang dijaga adalah ukuran SQLite. `sync_logs` di-cap 500.000 row (high) / 400.000 (low); `sync_sessions` di-cap 10.000 / 9.000. Eviction oldest-first via `id ASC LIMIT N` (autoincrement = proxy waktu, hemat index). Trigger: post-write async dari runner setelah `rows_failed` naik + `time.Ticker` 1 jam di `internal/maint/retention.go` sebagai safety net. **`auto_vacuum = INCREMENTAL` di-set di migrasi awal M1** (tidak bisa diubah retroaktif); `PRAGMA incremental_vacuum(N)` dipanggil setelah setiap eviction batch — tanpa ini file tidak shrink dan retensi gagal mencapai tujuannya. CASCADE: session ter-evict ikut menghapus `sync_logs` milik session tsb (termasuk `profile_snapshot_json`-nya). **Export log CSV wajib disediakan**: per-session di Sync Session detail + bulk di Settings/Health page; tombol manual "Bersihkan log lama" di-block sampai user klik export atau eksplisit confirm. Tidak ada export otomatis. UI Settings/Health menampilkan counter dan ukuran DB sebagai transparansi pengganti kontrol konfigurasi. Test boundary: 499.999/500.001 row + post-vacuum `page_count` reduction + CSV round-trip (UTF-8 BOM untuk Excel Windows).
- **Cancel UX**: user klik Cancel mid-sync → `POST /api/sessions/{id}/cancel` (idempotent) trigger `context.Cancel()` di runner. Cooperative cancel di chunk boundary per ADR-0003: **chunk yang sudah `BEGIN` selalu commit penuh** — Cancel hanya mencegah fetch chunk berikutnya, bukan abort transaksi yang sedang ter-execute. Tidak ada `tx.Rollback()` path di runner V1; chunk in-flight selesai normal (commit atau row-level error log via `ToFriendly`), lalu runner exit loop saat `ctx.Err() != nil` sebelum SELECT chunk berikutnya. Implikasi responsiveness: latency Cancel = sisa durasi chunk in-flight (chunk 500–1000 baris di network normal ≈ ratusan ms–detik); UI tampil spinner "Membatalkan…" antara klik dan SSE `event: cancelled`. Status final = `cancelled` (nilai kelima di enum `sync_sessions.status`: `running | done | interrupted | failed | cancelled`), beda semantik dari `interrupted` (system event: connection drop / app crash) dan `failed` (hard error). **Tidak ada rollback**: baris yang sudah ter-commit di Destination tetap ada — konsekuensi Source-wins UPSERT + chunk autocommit. UI menampilkan badge netral + pesan "{processed} baris sudah masuk Destination, hapus manual jika perlu". SSE `event: cancelled` payload `{processed, failed, cancelled_at}`. V2 mungkin tambah PK-tracking untuk rollback opsional (delete inserted rows saja, tidak undo overwrite) atau abort-chunk via `tx.Rollback()` untuk responsiveness lebih cepat.
- **Credential encryption**: AES-GCM untuk password koneksi MariaDB di SQLite. Hybrid `KeyProvider` interface di `internal/crypto/`: (a) OS keystore (go-keyring) sebagai default, (b) passphrase user + Argon2id KDF sebagai mode portable. Mode dipilih sekali di first-run wizard, disimpan di tabel `app_settings.key_mode` (`os_keystore` | `passphrase`); ganti mode = re-key flow (dekripsi semua credential dengan provider lama, re-enkripsi dengan provider baru). Lupa passphrase = credential hilang permanen (no recovery di V1). Salt + KDF params disimpan di `app_settings`, ciphertext + nonce di `connections.password_ciphertext`.
- **Mapping Builder UI (low-build-cost)**: granularitas **per-tabel via shadcn `<Tabs>`** + `<ScrollArea>` sidebar, satu tab = satu Destination table. Layout body = HTML `<Table>` standar dengan kolom `Kolom Destination | Sumber nilai | Detail | Status`. **Sumber nilai = satu shadcn `<Select>` grouped** (group "Kolom Source" + group "Opsi khusus": Konstanta, Kosongkan/NULL, Default DB, Lewati) — bukan two-step radio. Pilih "Konstanta" → reveal text input adjacent via conditional render. Auto-match feedback **hanya conditional className** (row putih + badge "Auto" abu-abu untuk auto-matched; row kuning muda + badge "Isi" oranye untuk unresolved) — tanpa animation library. Counter "X dari Y belum diisi" di header tab. **Drag-drop di-skip V1** (dnd-kit + a11y testing + edge case touch device terlalu mahal untuk junior dev / AI lokal kecil) — dropdown searchable cukup (cmdk bawaan shadcn Select). **Type-mismatch warning tidak ada di builder**: divalidasi di Preflight (single source of truth dengan schema drift detection, hindari duplikasi logic frontend↔backend). Tombol "Tambahkan aturan" di kolom Detail → buka shadcn `<Dialog>` (Rule editor terpisah dari builder utama). Stack: `Tabs`, `ScrollArea`, `Table`, `Select`, `Input`, `Badge`, `Button`, `Dialog` — semua shadcn standard, tidak ada custom UX. Trade-off yang diterima: scroll panjang untuk tabel ≥50 kolom; tidak ada visualisasi "kolom Source mana yang tidak terpakai"; feedback type mismatch baru muncul di Preflight, bukan saat user pilih sumber.
- **Mapping Profile lifecycle (`draft`/`ready`)**: kolom `mapping_profiles.status` dengan dua nilai. Validasi terbagi dua layer: **Builder-time** (`internal/repo/mapping_profiles.go` + UI) memvalidasi *structural impossibilities* — PK Destination wajib di-pair ke Source (bukan Constant/NULL/Default/Skip), PK tidak boleh punya Rule, kolom NOT NULL no-default tidak boleh "Kosongkan/NULL" atau "Lewati" (kolom NOT NULL ber-DEFAULT boleh "Lewati"), Closure Advisor expand dialog wajib, cross-profile collision dicek hanya terhadap profile lain yang `ready`. Layer ini blocking transisi `draft → ready` dengan error inline di kolom yang bermasalah. **Preflight** (`internal/sync/preflight.go`, saat Start) fokus pada *schema drift* — tidak mengulang aturan layer 1, asumsi profile sudah `ready`. Edit profile `ready` yang merusak invariant otomatis turun ke `draft` dengan dialog konfirmasi. `selection_json` di-freeze hanya saat `draft → ready` (Closure Advisor terlalu mahal untuk WIP save). UI: badge `Draft` (abu-abu) / `Siap` (hijau), tombol "Mulai Sync" hanya muncul di profile `ready`, builder header tampilkan counter "X dari Y kolom siap" + tombol "Tandai siap dipakai". `POST /api/sessions` dengan profile `draft` → 400 ramah. Snapshot di Sync Session tetap diambil saat session dibuat dari profile `ready`, sehingga edit mid-run (turun ke `draft`) tidak mengganggu runner aktif.
- **AUTO_INCREMENT di Destination**: V1 tidak menyentuh counter Destination. UPSERT memakai PK eksplisit dari Source via `INSERT ... ON DUPLICATE KEY UPDATE`; PK collision dari re-sync (baris berasal dari Source sebelumnya) lewat branch UPDATE — Source-wins. Collision yang **gagal** = UNIQUE non-PK (mis. email): error 1062 ditangkap `ToFriendly` dengan template "Baris bertabrakan dengan data yang sudah ada di Destination pada kolom unik {key} (nilai {val}). Kemungkinan ada baris di Destination yang bukan berasal dari Source", masuk `sync_logs`. Tidak ada `ALTER TABLE SET AUTO_INCREMENT` post-sync (lock cost + risiko bentrok dengan aplikasi lain yang pakai Destination), tidak ada pre-sync scan deteksi collision (mahal + tidak resolusi). User yang insert manual ke Destination dengan PK yang lalu di-claim Source: baris itu ter-overwrite tanpa peringatan — konsekuensi "Destination = derived copy".
- **Sync Log UI (group-by-default)**: Sync Session detail tampilkan row failures **default group-by `mariadb_code`** (mis. "8.231 baris — Referensi data hilang (1452)", "412 baris — Tipe data tidak cocok (1366)") via shadcn `<Accordion>`. Setiap group: count, ToFriendly summary, tombol "Unduh CSV grup ini", expand → tabel 50 baris pertama (Destination table, PK, kolom bermasalah, pesan ramah). Tombol "Lihat semua sebagai tabel" eskalasi ke flat paginated view (50/halaman, sortable kolom `created_at` + `mariadb_code`). Live count refresh via SSE `progress` event saat session running — group counts re-computed client-side dari snapshot + delta event. Tidak ada full-text search di pesan V1 (cost vs persona triage value rendah; CSV export cukup untuk handoff ke admin Source). Per-group CSV memakai filter `mariadb_code` di query, bukan filter client-side — hindari memori spike di failure set besar. UTF-8 BOM untuk Excel-Windows konsisten dengan ADR-0010 bulk export.
- **First-run onboarding (three-card progress)**: dashboard kosong tampilkan **tiga shadcn `<Card>`** berderet sebagai roadmap: (1) "Tambahkan Koneksi" — selalu unlocked di first-run; (2) "Buat Mapping Profile" — locked sampai `hasConnections` (≥1 Source + ≥1 Destination); (3) "Mulai Sync Pertama" — locked sampai `hasReadyProfile` (≥1 profile `status='ready'` per ADR-0014). Card locked tampil `disabled` (opacity 50%, tombol non-interaktif, tooltip "Lengkapi langkah sebelumnya dulu"). Card unlocked tampil CTA primer ("Tambah Koneksi", "Buat Mapping Profile", "Mulai Sync"). Copy per card maksimal 1–2 kalimat awam. Setelah `hasAnySession` true (sync pertama selesai sukses *atau* `done` dengan `rows_failed`), set tiga card dilepas dari dashboard — diganti daftar Sync Sessions terbaru + tombol "Mulai sync baru". Tidak ada library tour (Shepherd.js / Intro.js / Driver.js) — cost a11y + maintenance vs payoff persona target rendah. State derivation di server (single endpoint `GET /api/onboarding/state` return `{hasConnections, hasReadyProfile, hasAnySession}`); frontend tidak query 3 list terpisah. Trade-off yang diterima: user power yang sudah biasa flow tidak bisa skip ke "advanced view" — V1 default sederhana, advanced toggle V2.
- **Sample Preview (live inline per Rule editor)**: Rule editor dialog (per ADR-0012) menampilkan **panel preview live** di bawah form Rule — tabel dua kolom `Nilai Source → Hasil` dengan **5 nilai distinct + 1 baris NULL** sebagai sample. Sumber sample: `SELECT DISTINCT col FROM src.t WHERE col IS NOT NULL LIMIT 5` + 1 row hard-coded `NULL`, dijalankan saat dialog dibuka (cache per dialog session). Translasi Rule dijalankan **client-side** memakai output `internal/rules/translate.go` yang di-expose lewat endpoint stateless `POST /api/preview/rule` (body `{rule_dsl, source_connection_id, table, column}` → `[{source_value, dest_value, status: 'ok'|'error', error_friendly?}]`). Debounce **300ms** sejak user terakhir edit DSL — hindari spam request saat user mengetik regex. Per-row `status='error'` → cell `Hasil` tampil pesan ramah (mis. "Tipe tidak cocok") tanpa block save Rule (Rule boleh disimpan dengan preview error; runtime fallback per ADR-0003 yang menangani). Endpoint stateless: tidak menyentuh `mapping_profiles`, tidak persist sample value (PII concern di Source produksi). Loading state spinner di panel preview, tidak block tombol Save Rule. Tidak ada bulk preview "5 baris full row × all columns" di builder utama (cost: query × tabel besar × refresh per tab change terlalu mahal; persona target lebih butuh feedback per-rule fokus). Tidak ada preview untuk Sumber nilai non-Rule (Konstanta / Kosongkan / Default DB / Lewati) — nilai sudah eksplisit di Select.
- **Connection test (explicit Test+Save split)**: form Tambah/Edit Koneksi punya **dua tombol independen** — "Test Koneksi" (handshake `Open + Ping + USE db + SELECT 1`, timeout 5 detik per step, tidak persist) dan "Simpan" (persist apa adanya, tidak auto-test). Kolom `connections.last_test_at`, `last_test_status` (`untested`|`ok`|`failed`), `last_test_error_friendly` menyimpan hasil test terakhir; field credential berubah saat edit → reset ke `untested`. Endpoint: `POST /api/connections/test` (form payload, pre-save) dan `POST /api/connections/{id}/test` (post-save, update kolom). `ToFriendlyHandshake` map error → pesan Bahasa awam (1045 password salah, 1049 db tidak ditemukan, 2003 server tidak terhubung, 2002 timeout, 2005 host tidak dikenal); pesan **tidak** memuat password atau full DSN. UI list koneksi tampil badge `Belum dites`/`OK`/`Gagal` + tombol "Test Ulang"; Mapping Profile Builder banner kuning untuk koneksi `failed`, banner abu-abu untuk `untested` atau `last_test_at > 24h`. Banner advisory — tidak memblok save profile / start session (Preflight tetap catch handshake fail di chunk pertama). Threshold 24h hard-coded V1. Save yang hanya ubah label tidak reset `last_test_status`.
- **Lokasi internal SQLite (same-dir-as-binary)**: file `magicsync.db` (+ `.bak` self-heal) hidup di direktori yang sama dengan binary, di-resolve via `os.Executable()` + `filepath.EvalSymlinks` + `filepath.Dir()` di `internal/db/path.go`. Bukan CWD (rapuh terhadap double-click Windows / shortcut tanpa "Start in"), bukan `%APPDATA%`/`~/.local/share/`/`~/Library/Application Support/` (lepas janji portable USB-stick), bukan user-pilih saat first-run wizard (onboarding bloat). Symlink eval penting: `ln -s /opt/.../magicsync ~/bin/magicsync` → DB tetap di `/opt/...` (folder binary nyata) supaya pindah binary ikut bawa DB. Read-only folder (mis. binary di `/usr/local/bin/`) → bootstrap fail dengan pesan ramah stderr "Aplikasi tidak bisa menulis di folder X, pindahkan ke Desktop/Documents". Tidak ada native dialog di V1 (single-binary tanpa webview pop-up dependency). Distribusi M8 = satu file binary per OS + README, DB lahir saat first-run. Pindah lintas mesin = copy folder utuh; mode passphrase ADR-0011 ikut natural, mode OS keystore tetap terikat ke OS asal (banner re-input sudah ada). V2: env var `MAGICSYNC_DATA_DIR` override.
- **Sync Session retry (fresh-run-only)**: Session `interrupted` / `failed` / `cancelled` tidak bisa di-resume. UI Session detail untuk status non-`done` menampilkan tombol primer "Mulai Sync Baru" (bukan "Lanjutkan") yang redirect ke Start Session standar dengan profile sama pre-selected. Runner setiap Session traversal Selection Set dari index 0 dalam topological order; tidak ada lookup session sebelumnya, tidak ada resume state di `sync_sessions`. Idempotensi dijamin layer UPSERT (`INSERT ... ON DUPLICATE KEY UPDATE`) — baris cocok PK ter-UPDATE dengan nilai Source saat ini, baris baru ter-INSERT, baris yatim di Destination (Source delete sejak Session lalu) tetap ada (V1 tanpa delete propagation). Snapshot profile (ADR-0008) per Session independen — Session 2 freeze snapshot baru, tidak share dengan Session 1. Closure Advisor + Preflight re-jalan setiap Session start (no shortcut "ini resume, skip"). Trade-off: dataset besar (>10 juta baris) yang sering `interrupted` re-fetch full table dari Source — diterima sadar untuk simplisitas mental model + simplisitas runner. V2 candidate: `resume_from_table` + `resume_from_offset` di `sync_sessions`, delete propagation opt-in per profile, auto-retry exponential backoff untuk flaky network.
- **Single-session global concurrency**: maksimal **satu Sync Session `running`** di seluruh aplikasi pada satu waktu — bukan per-Profile, bukan per-Destination. `internal/sync/runner.go` pakai `sync.Mutex` package-level + `repo.SyncSessions.AnyRunning()` check di `POST /api/sessions` → return `409 Conflict` ramah `{error_friendly, conflict_session_id, conflict_profile_name}` kalau ada session aktif. Bootstrap (`internal/db/bootstrap.go`) mark row `status='running'` zombie (crash/SIGKILL session sebelumnya) jadi `interrupted` sebelum runner menerima request baru. UI: tombol "Mulai Sync" di profile non-aktif disabled dengan tooltip menyebut nama profile aktif; row pemilik session aktif tampil "Sedang berjalan…" + link ke Session detail; header global mini badge "1 sync berjalan". Tidak ada auto-queue (state machine tambahan tanpa value V1) — user explicit retry setelah session aktif selesai. Cancel mid-run release lock setelah chunk in-flight commit (per Cancel UX). Trade-off: power user dengan Source/Destination independent tidak bisa paralel — diterima sadar untuk persona UMKM. V2 candidate: per-Destination lock, auto-queue, multi-process file-lock. Lihat ADR-0020.
- **Distribusi M8 (Linux+Windows unsigned, manual cross-compile)**: V1 rilis untuk **linux/amd64 + linux/arm64 + windows/amd64**, **macOS skip** (Apple Developer Program $99/tahun prematur pre-revenue + Gatekeeper friction terlalu tinggi untuk persona UMKM). Cross-compile manual via `scripts/release.sh vX.Y.Z` di mesin dev (loop `GOOS`/`GOARCH` ~15 baris, output ke `dist/`), upload ke GitHub Release pakai `gh release create vX.Y.Z dist/* --notes-file CHANGELOG.md`. **Versioning SemVer longgar** mulai `v0.1.0` (per milestone progress) sampai `v1.0.0` (M1–M8 lulus + smoke 3-OS); pre-1.0 breaking change schema SQLite / profile JSON boleh tanpa MAJOR bump (V1 belum janji stabilitas). Versi di-inject via `-ldflags "-X main.version=$VERSION"`, di-expose lewat `GET /api/version` + UI Settings/About. **CHANGELOG.md** format [Keep a Changelog](https://keepachangelog.com/) Bahasa Indonesia audience user akhir (mis. "Tambah dukungan Sync Session yang bisa dibatalkan", bukan istilah teknis ADR); `gh release` membaca section terbaru. Flow rilis manual: edit `CHANGELOG.md` → `git tag vX.Y.Z` → `git push --tags` → `./scripts/release.sh vX.Y.Z` → `gh release create`. Tidak ada GitHub Actions di V1 — solo dev, cadence rendah, debug terminal lokal lebih mudah; public repo Actions tetap gratis kalau V2 butuh otomasi (reversible tanpa ADR baru, skrip yang sama bisa dipanggil dari workflow). Binary unsigned: Linux nihil friction, Windows SmartScreen "More info → Run anyway" didokumentasikan di README dengan 2 screenshot bahasa awam (tidak menyebut "code signing" / "EV cert"). Landing page badge platform + catatan ramah "macOS akan didukung versi mendatang, sementara bisa via Docker/Lima". `-ldflags "-s -w"` cukup; **tidak pakai UPX** (trigger AV false positive Windows Defender heuristic). V2 candidate: Apple Developer + notarization, Windows EV cert ($300–400/tahun) untuk hilangkan SmartScreen, Actions matrix on-tag-push, in-app auto-update via GitHub API. Lihat ADR-0021.
- **CSV export format (sync_logs)**: scope **per-Session** (bukan cross-Session) supaya `profile_snapshot_json` konsisten — admin Source yang menerima file paham aturan mapping yang berlaku saat run. **8 kolom fixed** dalam urutan: `waktu` (ISO 8601 UTC), `tabel_destination`, `pk_baris` (JSON object selalu di-quote, mendukung composite PK), `kolom_bermasalah`, `nilai_source` (string repr nilai yang ditolak; NULL = literal `NULL` tanpa quote), `kode_mariadb` (mis. `1452`, `1366`), `pesan_teknis` (raw MariaDB error), `pesan_ramah` (output `ToFriendly`). Tidak ada kolom `nilai_dest_existing` (Source-wins kontrak: nilai existing tidak relevan untuk triage), tidak ada `rule_yang_aktif` (bisa derive dari `profile_snapshot_json` di Session detail; menambahkannya = redundan + bocor DSL ke admin yang tidak butuh). **Encoding UTF-8 dengan BOM** (`\xEF\xBB\xBF`) supaya Excel Windows render Bahasa Indonesia karakter non-ASCII benar tanpa user pilih encoding manual. Quoting RFC 4180 (`csv.Writer` Go default): koma/newline/quote dalam field di-escape; `pk_baris` JSON selalu wrapped karena mengandung `{}` `:` `,`. **Streaming write** ke `io.Writer` (HTTP response) — tidak load full result set ke memory; failure set 100k+ baris tidak spike RSS. Filename: `magicsync-failures-{session_id}-{profile_slug}-{YYYY-MM-DD}.csv` (slug = lowercase + dash, max 40 char). Per-group CSV (tombol di Sync Log UI accordion) memakai SQL filter `WHERE session_id=? AND mariadb_code=?` di repo, **bukan** filter client-side — hindari memori spike. Bulk export di Settings/Health page mengeluarkan satu file per Session (bukan satu file gabungan). Tidak ada `.xlsx` (binary size cost untuk lib seperti excelize ~5MB tidak sebanding; user yang butuh Excel buka CSV langsung). Endpoint: `GET /api/sessions/{id}/logs.csv` dan `GET /api/sessions/{id}/logs.csv?mariadb_code=1452` (per-group). Content-Type `text/csv; charset=utf-8`, Content-Disposition `attachment; filename=...`.
- **App startup & lifecycle (ephemeral port + single-instance per data dir)**: V1 adalah **lokal-server-yang-buka-browser**, bukan webview. Startup flow: bootstrap SQLite (per ADR-0017) → acquire single-instance lock → `net.Listen("tcp", "127.0.0.1:0")` (kernel pilih ephemeral port; URL dibaca via `Listener.Addr()`) → start HTTP server → `pkg/browser.OpenURL(url)` → fallback print `Magic MariaDB Sync siap di {url}` ke stderr kalau auto-open gagal (headless, no DISPLAY, dll). **Bind 127.0.0.1 only**, bukan `0.0.0.0` — local-first security: kredensial MariaDB plaintext di memori runtime + endpoint mutating tanpa CSRF (V1 trust local boundary). Remote access = V2 explicit flag `--bind 0.0.0.0` + auth token. **Single-instance lock per data directory**: file `magicsync.lock` (advisory file lock via `github.com/gofrs/flock`) di folder yang sama dengan `magicsync.db`. Instance kedua di folder yang sama → baca URL dari lock file (server menulis port saat startup), `pkg/browser.OpenURL(url)`, exit 0 — user dapat tab baru ke instance yang sudah jalan, bukan error message. Folder berbeda = lock berbeda = independen (pakai mode passphrase + DB terpisah, sah jalankan dua instance simultan). **Tidak ada tray icon V1**: cross-OS systray lib (`fyne.io/systray`) menambah CGO + macOS skip per ADR-0021 + Windows COM init + Linux DBus dependency — biaya tidak sebanding dengan persona target yang model interaksinya episodic (buka, sync, tutup). Shutdown V1 = `Ctrl+C` di terminal / Task Manager / `kill PID`; README dokumentasikan "tutup jendela browser tidak menutup aplikasi — kembali ke terminal lalu Ctrl+C". Tidak ada idle timeout V1 (auto-shutdown setelah N menit tanpa request) — V2 candidate. Trade-off yang diterima: URL ganti tiap run (tidak ada bookmark stabil) — persona target double-click binary tiap sesi pakai, bukan bookmark URL; + user tanpa terminal awareness mungkin meninggalkan proses zombie sampai reboot (mitigasi: README jelas + V2 idle timeout). Lihat ADR-0022.
- **Timezone handling (UTC enforcement per koneksi)**: setiap koneksi MariaDB (Source + Destination) menjalankan `SET time_zone = '+00:00'` di `internal/mariadb/pool.go` setelah `Ping` sukses, sebelum query apapun — dijalankan via `db.SetConnMaxLifetime` aware connection-init hook (`mysql.Config.Params["time_zone"] = "'+00:00'"` di DSN, paling robust karena re-applied untuk setiap fresh connection di pool, termasuk yang reconnect setelah idle). **Implikasi `TIMESTAMP`**: kolom `TIMESTAMP` di MariaDB internal stored UTC, dikonversi ke session `time_zone` saat SELECT — UTC enforcement berarti nilai yang ditarik dari Source = nilai UTC apa adanya, ditulis ke Destination dengan `time_zone` yang sama = **identical wall-clock string**. Tidak ada drift implicit lewat zone session yang berbeda di Source (mis. `+07:00` Asia/Jakarta) vs Destination (mis. `UTC`). **Implikasi `DATETIME`**: kolom `DATETIME` adalah string wall-clock tanpa zone semantics — pass-through apa adanya, tidak terpengaruh `time_zone` setting. **Rule `date_format`**: parse memakai `time.ParseInLocation(layout, val, time.UTC)` untuk both `TIMESTAMP` dan `DATETIME` (layer `internal/rules/translate.go` tidak introspect tipe MariaDB — semua nilai diasumsikan UTC). Output reformat juga UTC. Edge case: kolom `DATETIME` yang user-edit dengan zone non-UTC (mis. wall-clock Jakarta `2026-05-02 14:00:00`) akan diperlakukan sebagai UTC saat di-format ulang — dokumentasikan di Rule editor tooltip "Waktu dianggap UTC". **Preflight handshake** menjalankan `SELECT @@session.time_zone, @@global.time_zone, @@system_time_zone` dan menyimpan ke `connections.notes_text` (advisory, non-blocking) — admin Source bisa lihat "session time_zone diforce ke +00:00 oleh aplikasi" kalau menyelidiki anomali laporan. Tidak ada UI per-koneksi timezone override V1 (kompleks: `IANA_NAME` tabel `mysql.time_zone_name` belum tentu loaded di MariaDB Source, fallback ke offset numerik confusing). Tidak ada handshake fail "Source dan Destination harus identik time_zone" — UTC enforcement membuat aturan itu redundan. **Trade-off diterima**: downstream pipeline yang baca `TIMESTAMP` dari Destination dengan `time_zone='+07:00'` akan lihat nilai bergeser 7 jam vs Source live (yang juga di-read dengan `+07:00`) — mitigasi: README docs section "Mengapa angka jam berbeda saat saya buka Destination dengan tools lain" + Settings/Health page tampilkan badge "Aplikasi memaksa zona UTC saat sync" sebagai transparansi. V2 candidate: per-koneksi timezone setting + Rule date_format dengan explicit `tz` field.
- **SQLite migration strategy**: plain numbered SQL files di `internal/db/migrations/` (`001_init.sql`, `002_add_x.sql`), embedded via `//go:embed migrations/*.sql`. Tabel `_migrations` (`version INTEGER PRIMARY KEY`, `applied_at TEXT`) track applied versions. Bootstrap (`internal/db/bootstrap.go`) loop: list embedded files, untuk version > `MAX(_migrations.version)`, eksekusi dalam transaction, insert version row. **No down migrations** V1 — rollback = self-heal `.bak` rebuild (file rename + recreate) atau user manual restore. **No third-party lib** (golang-migrate, goose, atlas) — single binary ethos, ~30 baris hand-rolled cukup. Pre-1.0 breaking schema change boleh tanpa MAJOR bump (ADR-0021 SemVer longgar); CHANGELOG.md wajib document migration yang affect existing data. Failure di tengah migration = transaction rollback, bootstrap exit dengan pesan ramah ke stderr, file `.bak` tidak dibuat (file aslinya tidak rusak — gagal sebelum apply, bukan corruption).
- **Connection pool sizing**: per koneksi MariaDB (Source + Destination) di `internal/mariadb/pool.go`: `SetMaxOpenConns(4)`, `SetMaxIdleConns(2)`, `SetConnMaxLifetime(30m)`, `SetConnMaxIdleTime(5m)`. Single-session global (ADR-0020) menjamin maksimal 8 connection aktif ke satu host saat sync (4 Source + 4 Destination — read+upsert+test). Aman untuk MariaDB default `max_connections=151`. Lifetime + IdleTime mencegah "MySQL has gone away" setelah server `wait_timeout` (default 8 jam, tapi sering di-tune lebih pendek di shared hosting). Hardcoded constant V1; UI tuning V2.
- **Charset & collation handling (DSN-enforce + runtime fallback)**: setiap koneksi MariaDB (Source + Destination) di-force `charset=utf8mb4` di DSN `internal/mariadb/pool.go` (`mysql.Config.Params["charset"] = "utf8mb4"`) — same location dengan UTC enforcement. **Tidak ada preflight detection** charset/collation per kolom: `INFORMATION_SCHEMA.COLUMNS.CHARACTER_SET_NAME` + `COLLATION_NAME` introspect mahal di skema lebar, dan persona target tidak akan reasoning hasil "tabel A.kolom B punya charset utf8mb3 sedangkan Destination utf8mb4 — apa yang harus saya lakukan?". `utf8mb3` deprecated di MariaDB 10.6+; `utf8mb4` Source → `utf8mb4` Destination = no-op (kasus terjauh paling umum). **Runtime per-row fallback** lewat existing `ToFriendly` whitelist untuk error 1366 (`Incorrect string value`): pattern matching pada hex prefix `\xF0\x9F` (4-byte UTF-8 leading byte = emoji / supplementary plane) → friendly hint "Nilai mengandung karakter khusus (emoji atau aksara non-Latin) yang tidak didukung kolom Destination. Minta admin Destination ubah charset kolom ke utf8mb4 dengan `ALTER TABLE {tabel} CONVERT TO CHARACTER SET utf8mb4`". Generic 1366 (non-emoji pattern) → existing template "Tipe data atau format nilai tidak cocok dengan kolom Destination". **Tidak ada collation mismatch detection V1**: skenario "Source `utf8mb4_general_ci` vs Destination `utf8mb4_unicode_ci` menghasilkan 1062 unique violation karena 'café' vs 'cafe' ranking berbeda" — dianggap rare untuk persona UMKM, existing 1062 friendly template ("Baris bertabrakan dengan data yang sudah ada di Destination pada kolom unik {key}") cukup mengarahkan triage. **Tidak ada Mapping Builder badge "charset mismatch"** — over-engineering untuk persona target, menambah noise visual yang tidak actionable di builder layer. Trade-off diterima: 1366 errors per-row baru muncul di `sync_logs` setelah first run (bukan upfront warning di Preflight) — mitigasi: friendly message guiding admin ke `ALTER TABLE` SQL command yang bisa di-paste langsung; user re-run setelah ALTER selesai (idempotent UPSERT membuat re-sync aman). V2 candidate: optional Preflight charset diff scan (per `INFORMATION_SCHEMA.COLUMNS`) sebagai non-blocking advisory, atau Mapping Builder per-kolom badge.
